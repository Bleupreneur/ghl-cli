# CLAUDE.md — ghl-cli

Guidance for Claude Code working **in this repo** (developing or maintaining it).

If you are an agent trying to **use** the CLI to call GoHighLevel APIs, stop here and read [`.claude/skills/ghl-cli/SKILL.md`](./.claude/skills/ghl-cli/SKILL.md) instead.

---

## What this project is

`@bleupreneur/ghl-cli` — a standalone TypeScript CLI that exposes the full GoHighLevel v2 public API (576 operations across 41 domains) as shell commands. It is generated from the official OpenAPI specs in `spec/`.

Authentication is **Private Integration Token only**. There is no Marketplace OAuth flow, no client id/secret, no refresh tokens, and no token broker. Keep it that way — see the *Scope* section of the README for why, and `docs/DESIGN.md` for the full reasoning. If a feature needs a hosted service to work, it does not belong in this CLI.

**One-liner:** `ghl contacts search`, `ghl search appointment`, `ghl raw GET /contacts/{id}`.

---

## The golden rule

**Never hand-edit `src/generated/operations.ts`.**

It is auto-generated by `scripts/gen.ts`. If a command name is wrong, a description is missing, or a parameter is mis-typed, fix the heuristic in `scripts/gen.ts` (or the upstream spec) and re-run `pnpm gen`. Any manual edit is overwritten on the next codegen run.

---

## Data-driven architecture

Everything flows from `spec/*.json`:

```
spec/*.json  →  pnpm gen  →  src/generated/operations.ts  →  Commander tree  →  GHL API
```

- `scripts/gen.ts` reads all 41 specs, resolves `$ref`s, extracts path/query/body params, picks the correct `Version` header, deduplicates command names, and emits a typed `Operation[]`.
- `src/buildCommands.ts` turns that array into a Commander command tree at startup.
- `src/params.ts`, `src/http.ts`, `src/auth.ts`, `src/output.ts` are entirely generic — every operation is handled the same way.
- Hand-written code lives only in `src/commands/{auth,raw,search,docs}.ts` — the four commands not derived from specs.

Adding new GHL endpoints = `pnpm sync-specs && pnpm gen && pnpm test`. No new command code.

## Commands

```bash
pnpm dev <args>       # run from source (tsx src/cli.ts) — no build needed
pnpm gen              # (re)generate src/generated/operations.ts
pnpm typecheck        # tsc --noEmit
pnpm test             # vitest run
pnpm lint             # biome check src scripts test
pnpm lint:fix         # biome check --write
pnpm build            # tsup → dist/cli.js
pnpm sync-specs       # refresh spec/ from GoHighLevel/highlevel-api-docs
```

Run `pnpm typecheck && pnpm test && pnpm lint` before every commit.

## Gotchas worth knowing

- **Root globals shadow subcommand flags.** `--profile`, `--api-key`, and `--location` are declared on the root program in `src/cli.ts`. Commander stores them on the root, not on the subcommand, even when the subcommand re-declares them. Any command that reads those three must use `cmd.optsWithGlobals()`, never `cmd.opts()`. `src/commands/auth.ts` and `src/buildCommands.ts` both do; `test/commands-auth.test.ts` pins the behaviour.
- **Env beats profiles.** `GHL_API_KEY` / `GHL_LOCATION_ID` sit above saved profiles in the resolution chain (`src/auth.ts`). An exported `GHL_API_KEY` makes `ghl auth use <name>` look broken. Worth remembering when a bug report says "it ignores my profile".
- **`resolveAuth` is synchronous.** It reads a file and returns; it does no I/O over the network. Do not `await` it.
- **Tokens must never be printed.** `redactKey()` in `src/auth.ts` is the only way a key reaches stdout.

## Coding conventions

- **ESM** throughout — `"type": "module"` in `package.json`.
- `import type` for type-only imports.
- **No `.js` extensions** in source imports (tsup resolves them at build time).
- **TypeScript strict mode.** No `any`; use `unknown` and narrow.
- **Biome** for linting and formatting (not ESLint or Prettier).
- Many small, focused files — keep each under ~300 lines.
- No secrets in source. Credentials live only in `~/.config/ghl/config.json` (chmod 600). Never log or print tokens.

---

## Testing

Vitest. Test files live in `test/`. Run `pnpm test` before committing.

Key areas: codegen validity (no duplicate command names, all operations mapped), params (CLI args → correct HTTP request), config CRUD, the auth resolution chain, `auth add` flag wiring, output modes, and the HTTP layer (auth headers, `Version` header, timeout, retry on 429).

---

## What this is NOT

- Not an OAuth client. No `auth login`, no client id/secret, no refresh tokens.
- Not a token broker or multi-tenant service.
- Not a client for GHL's private/internal APIs (site-builder, workflow-builder — those use Firebase auth).
